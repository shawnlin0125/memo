01

專案架構

採用 Monorepo 架構，每個 package 作為獨立微服務部署。Gateway 由 k8s Ingress 取代，不需獨立 package。

ticket-middleware/          ← 根 pom.xml（parent POM）
├── shared/
│   └── interfaces/         ← IProviderNode 介面定義（Maven module）
│       ├── pom.xml
│       └── src/main/java/
│           └── IProviderNode.java
├── packages/
│   ├── klook-node/         ← Klook 供應商實作（Maven module）
│   ├── kkday-node/         ← KKday 供應商實作（Maven module）
│   └── _template/          ← 新節點複製模板
└── infra/
    └── charts/provider-node/  ← Helm chart
注意 Gateway package 已移除

k8s Ingress + Service（CNAME）已涵蓋路由、認證、SSL 終止、流量限速、統一 log 等職責，不需額外的 gateway package。

02

分支策略與版本管理

單一主線 master，分支命名加節點前綴做視覺隔離。移除 development 與 release 分支，以 tag 控制版本與部署。

分支用途規則
master唯一主線，唯一部署來源只能透過 PR 合入；tag 觸發部署
feature/klook-{描述}-{task}功能開發完成後直接 PR 進 master
hotfix/klook-{描述}緊急修正從 master 開，修完合回 master

Tag 版本格式

Tag 永遠打在 master 上，代表「此節點此版本已審查可部署」。各節點版本號完全獨立。

klook-node/v1.0.0
klook-node/v1.1.0   ← patch: hotfix
klook-node/v1.2.0   ← minor: 新功能
kkday-node/v1.0.0
kkday-node/v2.0.0
版號規則

新功能 → minor 遞增（v1.1.0 → v1.2.0）
Hotfix → patch 遞增（v1.1.0 → v1.1.1）

03

開發流程

無論改動大小，所有修改一律透過 feature branch，確保 master 上每個 commit 都是經過測試的。

1

從 master 開 feature branch，命名加節點前綴

git checkout -b feature/klook-search-api-TASK-001
2

只改自己節點路徑下的檔案，push 觸發 CI 跑對應節點 scenario test

3

CI 通過後發 PR，review 後 merge 進 master

4

手動打 tag，CI 自動 build + deploy 對應節點

git tag klook-node/v1.2.0
git push origin klook-node/v1.2.0
04

部署流程

部署由 tag 名稱前綴觸發，與 commit 內容無關。只有打了對應 tag，CI 才會 build 並部署該節點。

feature/klook-xxx → PR → master merge
git tag klook-node/v1.2.0 && git push
CI 偵測 tag 前綴 refs/tags/klook-node/*
mvn build -pl packages/klook-node
部署 klook-node → production ✓
關鍵 kkday 完全不受影響

CI 由 tag 名稱決定 build 哪個子模組（-pl packages/klook-node），kkday 的程式碼就算在同一個 commit 裡也不會被 build 或部署。

05

多節點交錯合入情境

klook 與 kkday 同時有異動，klook 通過、kkday 不通過：

A(klook) → B(kkday) → C(kkday) → D(klook) → E(klook)
情況一 kkday 的 B、C 不影響 shared

直接在 E 打 klook-node/v1.1.0，CI 只 build packages/klook-node/。
kkday 修好後再補打 tag，兩者互不影響。

情況二 kkday 的 B、C 影響到 shared

必須先 revert B、C，確保 shared 是乾淨的，再打 klook tag。kkday 修好後重新 merge 並補打 tag。

git revert C
git revert B
# master 乾淨後再打 klook tag
git tag klook-node/v1.1.0
# kkday 修好後
git tag kkday-node/v2.0.1
原則 master 應隨時保持可部署狀態

不通過的節點應在 feature branch 的 CI 就被擋住，不應進入 master。若已合入且有問題，用 revert 而非讓壞程式碼停留。

06

Hotfix 流程

1

從 master 開 hotfix branch

git checkout -b hotfix/klook-fix-price-bug
2

修正 bug，push 觸發 CI 跑 klook scenario test

3

CI 通過 → PR merge 回 master → 打 patch 版 tag

git tag klook-node/v1.1.1   ← patch 遞增
git push origin klook-node/v1.1.1

與進行中 feature 的衝突

hotfix merge 進 master 後，如果此時有其他 feature PR 的 base 已落後：

# feature branch 需要先同步 master
git checkout feature/klook-search-api
git merge master   ← 解衝突（如果有）
git push           ← CI 重新跑一次

衝突發生機率低，hotfix 通常改動範圍小且精準，與進行中的 feature 動到同一行的情況不常見。

07

shared/interfaces 修改流程

shared/interfaces 存放 IProviderNode 共用介面，影響所有節點。修改時機：新增共用方法、修改方法簽章、新增共用資料結構（DTO）。

開發方式

單一 PR 包含 shared 改動與所有節點的實作更新，一次解決。

feature/shared-add-cancel-order
  ├── shared/interfaces/IProviderNode.java   ← 新增 cancelOrder()
  ├── packages/klook-node/...                ← 實作 cancelOrder()
  └── packages/kkday-node/...                ← 實作 cancelOrder()

CI 偵測到 shared/ 有變動，自動觸發所有節點的 test：

shared/** 有變動
  ├── 跑 klook scenario test
  └── 跑 kkday scenario test
      ↑ 全過才能 merge 進 master

降低衝擊：使用 default 方法

如果新方法不需要所有節點立即實作，可以提供預設實作讓各節點按節奏跟進：

// 舊方法不動
void searchTicket(String date);

// 新方法提供 default，不強迫立即實作
default void searchTicket(String date, String currency) {
    searchTicket(date);  // fallback 到舊方法
}

新增節點流程

1

執行腳本，自動從 _template/ 複製並替換佔位符

./scripts/new-node.sh my-new-vendor
2

腳本自動完成：

  • 複製 _template/ → packages/my-new-vendor-node/
  • 替換 pom.xml 內的 __NODE_NAME__ 佔位符
  • 建立 .github/workflows/my-new-vendor-node-ci.yml
3

實作 IProviderNode 介面，scenario test 骨架已在模板內

4

走標準開發流程：feature branch → CI → PR → master → tag → 部署

_template 內容

_template/
├── pom.xml                      ← artifactId 用 __NODE_NAME__ 佔位
├── src/main/java/
│   └── ProviderNodeImpl.java    ← 實作 IProviderNode 骨架
└── src/test/
    └── ScenarioTest.java        ← 標準 scenario test 骨架
09

CI/CD 自動化

路徑感知觸發（GitHub Actions 範例）

# .github/workflows/klook-node-ci.yml
on:
  push:
    paths:
      - 'packages/klook-node/**'
      - 'shared/**'   # shared 改動也觸發

jobs:
  test:
    steps:
      - run: mvn test -pl packages/klook-node

  deploy:
    needs: test
    if: startsWith(github.ref, 'refs/tags/klook-node/')
    steps:
      - run: ./infra/deploy.sh klook-node ${{ github.ref_name }}

品質閘門

階段動作通過條件
push feature branch跑對應節點 scenario test全數通過才可發 PR
PR merge 進 master再次跑 test（保險）通過才可 merge
master 打 tagbuild + deploy 對應節點只部署該節點
10

全自動化開發部署（AI 節點）

當節點由 AI 開發時，整個流程可高度自動化，人工介入降到最低。

1

人下指令：「新增 MyVendor 節點，支援搜尋、預訂、取消」

2

AI 自動執行：建立 feature branch → 複製 template → 實作介面 → 撰寫 scenario test → push

3

CI 自動跑 scenario test

  • 通過 → 自動發 PR
  • 不通過 → AI 讀 CI log 修正，重新 push，直到通過
4

人工確認 PR 內容，確認程式碼邏輯與測試情境符合業務需求

5

手動 merge + 打 tag，CI 自動部署到 production

人工介入的兩個點

PR review — 確認 AI 產出的邏輯正確，測試情境涵蓋真實業務場景，不只是讓 CI 通過。

打 tag — 部署到正式系統是高風險操作，由人決定何時部署。初期穩定後可考慮全自動。

11

共用資料庫與 Redis 架構

PostgreSQL 和 Redis 作為 pod 跑在同一個 k8s namespace，各節點透過 k8s Service name 直接連線，走內部網路，不需要外部主機。

k8s namespace 架構

namespace: ticket-middleware
  ├── klook-node pod
  ├── kkday-node pod
  ├── postgresql pod     ← 同 namespace，內部通訊
  └── redis pod          ← 同 namespace，內部通訊

資料隔離方式

資源隔離方式範例
PostgreSQLschema 隔離，各節點只存取自己的 schemaklook.*、kkday.*
Rediskey prefix 隔離klook:、kkday:

Spring Boot 設定

同一個 DB instance，各節點透過 default_schema 指定自己的 schema，連線字串用 k8s Service name：

# klook-node application.yml
spring:
  datasource:
    url: jdbc:postgresql://postgresql:5432/ticket_middleware
  jpa:
    properties:
      hibernate:
        default_schema: klook   ← 指定自己的 schema
  redis:
    host: redis               ← k8s Service name
    port: 6379

Flyway — 資料庫版本控制

Flyway 是資料庫的 Git，管理 schema 的變更歷史。節點啟動時自動執行還沒跑過的 migration，確保每個環境的 schema 一致，不需手動進 DB 下 SQL。

# migration 執行順序
V1__init_schema.sql       ← 建立初始 table
V2__add_currency_col.sql  ← 新增欄位
V3__add_index.sql         ← 新增 index

新節點的初始 migration 放在 _template 裡，new-node.sh 複製時自動替換佔位符：

-- V1__init_schema.sql
CREATE SCHEMA IF NOT EXISTS __NODE_NAME__;
SET search_path TO __NODE_NAME__;

CREATE TABLE orders (...);
CREATE TABLE tickets (...);
好處 共用 infrastructure，各自隔離資料

不需自己管主機和資料庫伺服器，統一由 k8s 管理備份、擴展、監控。同 namespace 內部通訊延遲低，且各節點資料透過 schema 與 key prefix 完全隔離互不干擾。
