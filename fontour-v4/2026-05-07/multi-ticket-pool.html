03 · Zero Downtime Deployment

零停機更新策略

每個 Provider Node 獨立容器部署，更新 klook-node 時 kkday-node 完全不受影響。核心機制為 Rolling Update + Readiness Probe + Graceful Shutdown 三位一體。

Rolling Update 配置

# kubernetes deployment 範例（每個 Provider Node 套用）
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1        # 最多多起一個新 Pod
    maxUnavailable: 0  # 不允許任何服務中斷

# 邏輯：先起新容器 → 確認 Readiness 健康 → 才逐步替換舊容器

Readiness vs Liveness Probe（健康檢查分工）

探針類型 問題 失敗行為 關鍵用途
/health/live 程序還活著嗎？ 重啟容器 防止殭屍程序佔用資源
/health/ready 已準備好接流量嗎？ 不給流量（但不重啟） 零停機更新的關鍵 — 新容器未就緒前不收任何流量

Readiness Probe 實作（NestJS）

// 每個 Provider Node 必須實作 /health 端點
@Get('/health/live')
liveness() {
  return { status: 'ok' };
}

@Get('/health/ready')
async readiness() {
  // 確認 Redis 可讀寫、外部供應商 API 可連線，才回傳就緒
  await Promise.all([
    this.redisClient.ping(),
    this.providerClient.ping(),   // 確認外部 API 連線
  ]);
  return { status: 'ready' };
}

優雅關閉（Graceful Shutdown）流程

SIGTERM 收到
→
停止接受新請求
→
等待進行中請求完成
→
釋放資源，容器退出
NestJS: app.enableShutdownHooks() · K8s: terminationGracePeriodSeconds: 30
04 · Fault Isolation

故障隔離設計

從容器層到應用層的雙重隔離，確保單一供應商異常不會雪崩影響其他服務。

Circuit Breaker — 熔斷器狀態機

🟢
CLOSED（正常）
直接呼叫外部供應商 API，計數連續失敗次數
失敗超過閾值 →
🔴
OPEN（熔斷）
直接回傳 Redis 快取或錯誤，完全不打 API，避免雪崩
← 冷卻期後 →
🟡
HALF-OPEN（試探）
試探性發送少量請求，成功則關閉熔斷器，失敗則再次熔斷
→ 成功則 CLOSED
建議整合 opossum 函式庫 · 每個 Provider Node 獨立熔斷器實例，互不干擾

Bulkhead — 艙壁隔離

為每個 Provider 設置獨立的連線池與請求佇列，避免某一供應商 API 響應緩慢時，耗盡全域連線資源並拖垮其他節點。

隔離層面 設計方式 防護效果
容器層隔離 每個 Node 獨立 K8s Deployment 程序崩潰不影響其他 Node
連線池隔離 各 Node 獨立 HTTP/Redis 連線池 慢速供應商不耗盡共享連線
請求佇列隔離 各 Node 獨立 Queue 限制並發 防止一個 Node 的積壓蔓延
熔斷器隔離 每個 Node 獨立 opossum 實例 Klook 熔斷不影響 KKday 運作
05 · Standard Interface Contract

票池節點標準介面規範

每個 Provider Node 必須繼承並實作以下標準介面，實現「插拔式」架構。新增供應商只需實作此介面，Gateway 和主系統零變更。

GET
search()
查詢可用性與即時價格。結果暫存於 Redis Price Buffer，避免重複打外部 API。
POST
createOrder()
在外部供應商端建立佔位訂單，回傳 External_Provider_ID 並寫入 Mapping Table。
PUT
confirmPayment()
確認支付並執行正式出票，完整紀錄 Request/Response 至 Audit Log (JSONB)。
DEL
cancel()
處理取消與退款請求，呼叫供應商退款 API 並同步更新主系統訂單狀態。

資料儲存與對帳機制

儲存組件 目的 技術實作
Mapping Table Internal_ID ↔ External_Provider_ID 對應 PostgreSQL 結構化欄位
Audit Logs 每次 Provider API 完整 Request/Response 記錄，供爭議排查 PostgreSQL JSONB 格式
Price Buffer 暫存供應商報價，改善搜尋體驗、減少外部 API 呼叫 Redis，設 TTL 自動失效

07 · Priority Actions

最優先落地的三件事

依照「成本低、效益高、風險可控」原則排序，建議依序執行。

Readiness Probe + 優雅關閉
零停機更新的基礎建設，成本低、效益高。只需實作 /health/live 和 /health/ready 端點，配合 Kubernetes terminationGracePeriodSeconds: 30，即可在不中斷用戶服務的情況下隨時更新任意節點。
Circuit Breaker（熔斷器）
防止單一供應商 API 異常拖垮整個票池服務。整合 opossum 函式庫，為每個 Provider Node 配置獨立熔斷器，當外部 API 連續失敗時自動切換至快取模式，保護整體系統穩定性。
結構化 JSON 日誌
配合 4.2 的 Audit Logs，統一輸出格式並接入監控系統（如 ELK Stack）。結構化日誌不僅用於爭議排查與對帳，更是後續導入 Tracing 與 Metrics 告警的基礎。成本最低、回報最快。
