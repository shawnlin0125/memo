Saga 是一種處理跨步驟、跨服務失敗補償的設計模式。
傳統 DB Transaction 失敗可以 ROLLBACK，但一旦跨越了 Redis、Kafka、外部 API，就沒有統一的 rollback 機制。
Saga 的核心概念是：每一個正向步驟，都要預先設計一個對應的「反向補償操作」，失敗時從斷點往回依序執行補償，代替 rollback。
🗄️ 傳統 DB Transaction
所有操作在同一個 Transaction 內，任一步驟失敗，DB 引擎自動 ROLLBACK 所有已執行的變更。

前提：所有操作都在同一個 DB 連線內，不跨服務。
BEGIN INSERT orders ... ✓ UPDATE inventory ... ✓ INSERT invoice ... ✗ 失敗 ROLLBACK ← DB 自動回到起點
🔁 Saga Pattern
每個步驟獨立執行，沒有統一 rollback。失敗時，靠預先定義的補償操作手動逐步回退。

前提：操作跨越 Redis、Kafka、外部 API 等，無法共享 Transaction。
步驟1 成功 → Redis 預扣庫存 步驟2 成功 → Kafka Produce 步驟3 成功 → DB 建立訂單 步驟4 失敗 → 折價券扣款失敗 ↓ 補償3 → DB 刪除已建訂單 補償1 → Redis 退還預扣庫存 通知 → 告訴用戶下單失敗
本系統的 Saga 補償流程（Consumer 失敗時）
📌 重要原則：補償要「從哪裡斷，從哪裡退」。步驟3失敗，只需退步驟1和2，不需要退步驟4（還沒執行到）。每個補償操作本身也需要冪等設計。

設計原則與注意事項
✅ 補償操作必須冪等：補償本身也可能失敗後重試，例如「退還 Redis 庫存」執行兩次，不能讓庫存被多退。
✅ 折價券扣款放最後：越晚扣，失敗時需要補償的範圍越小。
✅ 保留失敗記錄：建議將 Payment 標記為 FAILED 而不是刪除，保留稽核軌跡，方便排查問題。
✅ 補償失敗怎麼辦：補償操作本身也可能失敗（例如 Redis 掛掉），需要 dead letter queue 或 alert，轉人工處理。這是 Saga 無法完全避免的最終邊界。
針對「完整 DB 前置驗證 → Redis 預扣 → Kafka → Consumer 建單」架構，結合現有系統特性的主要風險點。
⚡
Redis 預扣成功，Kafka Produce 失敗
高
Redis 已扣庫存但事件沒進 Kafka（網路抖動、Kafka 暫時不可用），Consumer 永遠不會收到，庫存白白佔用直到 TTL 過期。
對策： Kafka Produce 失敗時立即同步回退 Redis 預扣（Lua 反向操作），整個同步段視為失敗回應用戶，靠 TTL 作最後保底。
📦
Kafka Payload 缺少欄位導致建單失敗
高
Consumer 不再回查 DB，完全依賴 Kafka payload 建單。若 Produce 時漏帶欄位，或日後欄位新增但 payload schema 未同步更新，Consumer 拿到的資料不完整就會建單失敗。
對策： 使用 Schema Registry（例如 Confluent Schema Registry + Avro / JSON Schema）管理 payload 格式；Produce 端加 schema 驗證，欄位不完整就不 Produce。
🔄
Consumer 重試造成重複建單
高
Kafka at-least-once 語意，Consumer 崩潰後重啟同一訊息再次消費，若建單非冪等，產生兩張 Payment，造成重複扣款或重複發票。
對策： 以 paymentNo 為冪等 key，Consumer 建單前先查 DB。資料庫加 unique constraint 作最後防線，捕捉 duplicate key exception 視為成功跳過。
⏱️
Consumer 建單超過 Redis TTL
中
Consumer 積壓嚴重，消費到這筆時預扣 key 已過期，Redis 自動釋放庫存。若 Consumer 不驗 TTL 就繼續建單，會建出一張沒有對應庫存扣減的訂單。
對策： Consumer 建單前先 ZSCORE 確認預扣 key 仍存在，不存在則走 FAILED 流程通知用戶。TTL 設定需大於 Consumer 最大可能 lag（需壓測評估）。
🏗️
BookingSetting Mutex 初始化 Race Condition
中
現行 ZADD Mutex 是多步非原子操作。大流量下多請求同時到，若 bookingSettingId 尚未初始化，放進預扣 Lua 前需重新設計，否則初始值可能被競爭覆蓋。
對策： 將初始化邏輯整合進預扣 Lua Script，以 SET NX 替代 ZADD Mutex，Lua 內原子完成「初始化判斷 → 設初始值 → ZINCRBY 預扣」三步。
📊
Consumer 積壓，PENDING 過久
中
大流量下 Consumer 處理不及，訂單長時間停留 PENDING，超過用戶心理預期造成客訴和重複下單。
對策： Consumer partition 數可水平擴展；設 PENDING 超時機制（e.g. 5 分鐘後自動 FAILED 並退 Redis 庫存）；監控 Kafka consumer lag，超閾值自動告警。
🔵 同步下單（現行）
繼續維持的理由
  • 邏輯集中，Debug 快，工程師認知負擔低
  • 訂單狀態立刻確定，用戶體驗直覺
  • 折價券、庫存、發票一致性有保障
  • 不需 Kafka、Consumer、Saga 維運成本
  • 高峰 QPS 受 DB W 連線數制約，難爆發
  • 每筆下單 RT 隨流量劣化
  • JVM Lock 僅保護單節點，多節點需再處理
🟢 非同步下單（新方案）
導入的理由
  • 前置驗證完整保留，可下單條件與現行相同
  • DB W 全部移出關鍵路徑，吞吐量大幅提升
  • Redis Lua 原子預扣，防超賣能力強
  • Consumer 可水平擴展，處理速度彈性調整
  • 用戶必須接受「建單中」狀態，前端需改動
  • Kafka payload schema 設計要一次做好，日後演進成本高
  • Saga 補償、冪等、TTL 監控，工程複雜度上升

📌 建議：以此方案為目標，分階段安全導入

此版本（保留完整 DB 前置驗證 + 建單本體非同步）是三個方案中最平衡的選擇：正確性不打折，DB 寫入壓力大幅降低，比「完全不查 DB」的極端版本更可控。

第一步（低風險，先做）：設計並確認 Kafka payload schema，確保所有建單欄位都能從 payload 還原，不需要 Consumer 再回查 DB。這是整個方案的地基。

第二步（核心）：實作預扣 Lua Script（含 BookingSetting 初始化原子化）+ Consumer 冪等機制（paymentNo unique constraint）。

第三步（特定活動先試）：以 Feature Flag 控制，對高峰活動啟用非同步路徑，其餘一般下單仍走同步。出問題可立即回切。

必須在上線前完成的安全網：① Produce 失敗的 Redis 即時回退、② PENDING 超時自動釋放庫存、③ Consumer lag 監控告警、④ Redis 與 DB 庫存差值對帳 job。
