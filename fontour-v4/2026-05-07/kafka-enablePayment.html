新方案移除了 Redis 安全網與背景 Job，改用 Consumer 內明確的 Saga 流程作為唯一補償機制。
關鍵設計：外部 API 呼叫放在所有 DB 寫入之前。外部 API 失敗 → 只需退款，DB 尚未寫入無需回退；DB 失敗 → 只需 Rollback，外部已啟用無需退款。兩個補償域完全隔離，複雜度大幅降低。

同步路徑例外時，DB Rollback，Redis 安全網 key 殘留等背景 Job 掃描退款。補償時機不確定，Job 邏輯需另外維護。

本系統的 Saga 補償流程（Consumer 失敗時）
📌 核心設計原則：外部 API 先於所有 DB 寫入。外部 API 失敗時 DB 尚未動，補償只需退外部款；DB 失敗時外部已啟用，補償只需 Rollback DB。兩個失敗域不重疊，補償邏輯清晰。

設計原則與注意事項
✅ 外部 API 先於 DB 寫入：外部失敗時 DB 無需回退；DB 失敗時外部無需退款。補償域隔離是整個方案複雜度可控的關鍵。
✅ 退款補償也需冪等：退款補償失敗重試時，先查 externalStatus 是否已是 REFUNDED，避免重複退款。
✅ 保留失敗記錄：Payment / PTO 標記 FAILED 而非刪除，保留完整稽核軌跡。
✅ DLQ 作為最終保底：若 Saga 補償（退款 API）也失敗，送 DLQ + 立即告警，帶完整 context（paymentId、已成功 provider 清單），轉人工跟進。
✅ 移除 Redis 安全網與背景 Job：補償邏輯集中在 Consumer 內，消除了「Job 掃描時機不確定」與「Redis key 殘留監控」的維運負擔。
針對「B2B 餘額門禁 → Kafka Produce → Consumer（外部 API → DB 寫入 → Redis 更新）」架構的主要風險點。
🔄
外部 API 重複啟用（冪等缺失）
高
Consumer 重啟或 Saga 補償重試時，可能重複呼叫外部 API（THSR 重複出票、Klook 重複核銷），導致用戶被重複扣款或拿到多張票，且部分外部系統不支援冪等鍵。
對策：呼叫外部 API 前，先查 externalOrder.externalStatus；若已是 ENABLED 則跳過。對支援冪等 key 的 provider 傳入 paymentId 作為 idempotency key。
📦
Kafka Payload 缺少欄位（業務欄位多）
高
enablePayment 業務欄位極多（商品類型、物流設定、折價券、信託類型、B2B 標記、apiProvider 清單），任何遺漏導致 Consumer 無法正確建立記錄，且 payload 缺欄位無法靠重試修復。
對策：使用 Schema Registry（Avro / JSON Schema）管理格式；Produce 前加 schema 驗證，欄位不完整則不 Produce；Consumer 收到不完整 payload 送 DLQ 而非重試。
🌐
退款 API 也失敗（補償失敗）
高
Saga 補償呼叫退款 API 時，外部服務同樣不可用，退款無法完成。用戶外部訂單已啟用但 Payment 為 FAILED，形成資料不一致且無法自動恢復。
對策：退款補償設指數退避重試；重試耗盡後送 DLQ + 立即告警，保留完整 context（paymentId、已成功 provider 清單）轉人工跟進。
⏱️
「處理中」狀態過久，用戶重複點擊
中
Consumer 積壓或外部 API 重試時，Payment 長時間無 DB 記錄。用戶查詢找不到，可能誤以為失敗而重複點擊，產生多筆 Kafka 訊息，Consumer 冪等壓力增加。
對策：Kafka produce 成功後將 paymentId 存入 Redis（TTL 30 分鐘）作為「處理中」標記，前端查狀態先查 Redis；設「處理中」超時告警，超 15 分鐘未完成即人工檢視。
🎫
TicketPool / ExternalRedeemCode 取票競態
中
多個 Consumer 同時處理不同訂單時，可能搶到同一張 TicketPoolOrder 或 ExternalRedeemCode（取「最早到期可用票」邏輯），導致 unique constraint 衝突或重複標記 STATUS_USED。
對策：在取票環節使用 SELECT FOR UPDATE（悲觀鎖）或 DB unique constraint 捕捉衝突後重試取下一張；或改用 Redis RPOP 作為票池佇列（原子取票，無競態）。
📊
Consumer 積壓觸發外部 API 重試風暴
中
高峰期 Consumer 積壓時，大量訊息同時開始呼叫外部 API，可能觸發外部服務限流（rate limit），造成雪崩式重試失敗，反而讓外部服務更難恢復。
對策：重試加入隨機 jitter 避免同時重試；監控 Consumer lag，超閾值時暫停部分 partition 降流；事先與各外部 provider 確認 rate limit 規格，Consumer 並發數不超過上限。
🔵 同步啟用（現行）
繼續維持的理由
  • 邏輯集中，一條 call stack，Debug 直覺
  • 票券狀態立即確定，用戶取票無等待
  • 外部 API 例外處理（逐 provider 退款）邏輯已成熟
  • 不需 Kafka、Consumer、Saga 等額外架構
  • 外部 API 延遲直接阻塞 Thread，高峰體驗差
  • 外部服務抖動時整體 RT 劣化，難水平擴展
  • paidPayment DB 早於外部 API，失敗需靠 Rollback，Redis 已更新無法自動回退
🟢 非同步啟用（新方案）
導入的理由
  • 同步段 < 50ms：無 DB 寫入、無外部 API，RT 極低
  • 外部 API 完全解耦，抖動由 Consumer 重試吸收
  • 外部 API 成功後才寫 DB，一致性結構比現行更正確
  • Saga 補償明確，消除 Redis 安全網與背景 Job 的維運負擔
  • 票券延遲顯示，前端需實作「處理中」狀態
  • 外部 API 冪等設計複雜（部分 provider 不支援冪等鍵）
  • Kafka payload 欄位多，schema 管理與日後演進成本高

📌 建議：以「外部 API 與 DB 寫入完全隔離」為核心，分階段安全導入

此方案最關鍵的設計決策是外部 API 呼叫放在所有 DB 寫入之前。這讓補償域完全隔離：外部失敗只退款、DB 失敗只 Rollback，比現行「DB 已寫入後外部失敗需靠 @Transactional Rollback + Job 補償」的混合補償更簡單、更可靠。

第一步（必做，上線前地基）：設計 Kafka payload schema，枚舉所有商品類型（SmartCard / ExternalRedeemCode / TicketPool / DELIVERY）與所有 apiProvider 所需欄位；確認 Consumer 可從 payload 完整還原所有處理路徑，不需回查 DB。使用 Schema Registry 管理格式演進。

第二步（核心安全機制）：實作外部 API 冪等查詢（externalStatus 判斷）、Consumer 冪等（paymentId + DB 狀態查詢）、指數退避重試（含 jitter）、DLQ + 告警流程，以及「處理中」狀態的 Redis TTL 標記。這些機制必須在功能上線前全部就緒。

第三步（特定 provider 先試）：以 Feature Flag 控制，對外部 API 延遲最嚴重、且退款邏輯相對簡單的 provider 先啟用非同步路徑（例如單一 provider 的訂單），確認 Consumer / Saga 穩定後再全量開放。

必須在上線前完成的安全網：① 外部 API 冪等查詢（防重複啟用）、② 退款補償指數退避 + DLQ 告警、③ Consumer lag 監控告警、④「處理中」Redis TTL + 超時告警（15 分鐘）、⑤ 壓測確認 Consumer 並發數不觸發各 provider rate limit。
