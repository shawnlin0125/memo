01

什麼是 Apache Kafka？

起源

由 LinkedIn 於 2010 年開發，2011 年捐給 Apache，用於解決每天處理數十億條事件的挑戰。名稱致敬作家法蘭茲·卡夫卡（Franz Kafka）。

核心定位

分散式、高吞吐量的事件串流平台（Event Streaming Platform），同時具備訊息佇列（Message Queue）與資料管道（Data Pipeline）功能。

三大核心能力

Publish / Subscribe — 發布與訂閱事件串流
Store — 持久化儲存事件，支援容錯
Process — 即時或歷史事件處理

關鍵特性

高吞吐量（每秒百萬級訊息）· 低延遲（毫秒級）· 水平擴展 · 持久化儲存 · 容錯機制 · 支援精確一次語義（Exactly-once）

核心思維：Kafka 把資料視為無限的 Log（日誌），Producer 不斷往 Log 尾端 append 資料，Consumer 從特定位置讀取，兩者完全解耦。這種設計讓 Kafka 同時支援實時串流與歷史重播（Replay）。

重要觀念：同一份 Topic 資料可以同時被多個 Consumer Group 讀取，各自維護獨立的 offset，互不干擾。這是 Kafka 相較傳統 MQ 的核心優勢之一。
03

核心元件詳解

Broker
訊息代理節點
Kafka Cluster 中的單一伺服器節點，負責接收、儲存並傳遞訊息。每個 Broker 有唯一的 ID，Cluster 通常由 3~10 個 Broker 組成以確保高可用性。
  • 每個 Broker 持有部分 Partition（Leader 或 Replica）
  • 一個 Broker 失效時，其他 Broker 的 Replica 接手成為新 Leader
  • Bootstrap Broker：Producer/Consumer 連接的入口點
  • Kafka 3.0+ 不再需要 ZooKeeper，改用 KRaft 模式
Topic
主題 / 訊息類別
邏輯上的訊息分類容器，類似資料庫中的 Table。Producer 將訊息發布到特定 Topic，Consumer 從 Topic 訂閱讀取。名稱為字串，需唯一。
  • 一個 Topic 可以有多個 Partition（平行處理）
  • Topic 是 append-only 的不可變 Log
  • 支援設定 Retention 保留政策（時間/大小）
  • 可設定 Replication Factor 控制副本數量
Partition
分區
Topic 的實體儲存單位，是 Kafka 實現水平擴展與並行處理的關鍵。每個 Partition 是有序的、不可變的訊息序列，Partition 數量決定最大並行消費數。
  • 每個 Partition 有一個 Leader 和多個 Replica（ISR）
  • 訊息在 Partition 內部保證有序，跨 Partition 不保證
  • Producer 用 Key Hash / Round-Robin 決定寫入哪個 Partition
  • Partition 數量增加可提高吞吐量，但增加後無法減少
  • 規劃原則：Partition 數 ≥ Consumer 數（同一 Group）
Consumer Group
消費者群組
一組共同消費 Topic 的 Consumer 集合，共享同一個 Group ID。Kafka 保證同一 Group 中每個 Partition 只被一個 Consumer 讀取，實現負載均衡。
  • Group 中的 Consumer 數 > Partition 數時，多餘的 Consumer 閒置
  • Group Rebalance：Consumer 加入/離開時重新分配 Partition
  • 不同 Group 獨立消費，互不影響
  • Group Coordinator 負責管理 Group 成員關係
Offset
偏移量
每條訊息在 Partition 中的唯一遞增序號，從 0 開始，不可變更。Consumer 透過追蹤 Offset 確認讀取進度，支援重播（Replay）歷史訊息。
  • Offset 儲存在特殊 Topic：__consumer_offsets
  • Auto-commit vs Manual-commit（避免訊息遺失或重複）
  • 可重設 Offset 到指定位置，實現訊息重播
  • earliest / latest / timestamp 三種重設策略
Producer / Consumer
生產者 / 消費者
Producer 負責發布訊息到 Topic，可指定 Key 決定路由 Partition，支援 acks 設定（0/1/all）控制可靠性。
Consumer 訂閱 Topic 並從 Partition 拉取（Poll）訊息。
  • acks=0：不等確認（最快，可能丟失）
  • acks=1：Leader 確認即可（預設）
  • acks=all：所有 ISR 確認（最安全）
  • Consumer 採 Pull 模式（非 Push），自行控制速率
Replication & ISR（In-Sync Replicas）
複製機制與同步副本集合
Kafka 透過 Replication Factor 決定每個 Partition 複製幾份。ISR 是當前與 Leader 保持同步的 Replica 集合。只有 ISR 中的 Replica 才有資格被選為新 Leader。
# 建立 Topic 時指定 Replication Factor kafka-topics.sh --create \ --topic user-events \ --partitions 6 \ --replication-factor 3 \ --bootstrap-server localhost:9092 # 建議：replication.factor >= 3，min.insync.replicas = 2
04

保留政策（Retention Policy）

⏱
時間保留

依照訊息的存活時間決定是否刪除，預設保留 7 天（168 小時）

log.retention.hours=168 log.retention.ms=604800000
💾
大小保留

依照 Log 總大小限制，超過上限時刪除最舊資料

log.retention.bytes=1073741824 # 預設 -1（不限制）
🔑
Log Compaction

依 Key 保留最新一筆訊息，舊版本刪除，適合狀態型（State）資料

cleanup.policy=compact # 或同時開啟 cleanup.policy=compact,delete
Log Compaction 運作原理

Compaction 不是即時刪除，而是由後台執行緒定期掃描並合併相同 Key 的訊息：

政策 觸發條件 適用場景 注意事項
Delete（時間） 超過 retention.ms 一般事件日誌、不需要歷史全量 以 Segment 為單位刪除
Delete（大小） 超過 retention.bytes 磁碟空間有限的環境 時間和大小可同時設定（OR 邏輯）
Compact 後台週期性執行 使用者設定、資料庫 CDC 快照 Key 為 null 的訊息不會被保留
Compact + Delete 兩者皆觸發 需要最新狀態且控制空間 Tombstone 訊息（值為 null）可刪除 Key
05

訊息完整流程

Producer 發送流程
1
建立 ProducerRecord
包含 Topic、可選的 Key/Value/Partition/Timestamp
2
Serialization 序列化
Key/Value 轉換為 bytes（JSON、Avro、Protobuf）
3
Partitioner 路由
有 Key → Hash 到固定 Partition；無 Key → Sticky/Round-Robin
4
RecordAccumulator 批次緩衝
批量傳送以提高吞吐量（batch.size / linger.ms 控制）
5
Sender 發送 + 等待 ACK
依 acks 設定決定可靠性，失敗依 retries 重試
Consumer 消費流程
1
加入 Consumer Group
向 Group Coordinator 註冊，觸發 Partition Rebalance
2
Poll 拉取訊息
consumer.poll(timeout) 向 Leader Partition 拉取批次訊息
3
Deserialization 反序列化
bytes 轉回業務物件，需與 Producer 格式一致
4
業務邏輯處理
處理訊息內容，寫入 DB 或觸發下游流程
5
Commit Offset
提交已處理進度，Auto-commit 或手動提交（更安全）
06

業界實際應用方式

📊
實時資料管道（Data Pipeline）
LinkedIn · Airbnb · Uber

串接多個系統，讓資料從 DB/App → Kafka → Data Warehouse / Data Lake。使用 Kafka Connect 減少自製 Connector 的工作量，支援 Source 和 Sink Connector。

🏦
金融交易事件處理
Morgan Stanley · PayPal · 玉山銀行

處理轉帳、付款事件串流，結合 Kafka Streams 進行即時詐騙偵測（Fraud Detection）。Exactly-once 語義確保每筆交易不重複處理。

🛍
電商行為追蹤
Shopee · Lazada · 蝦皮

追蹤使用者點擊、加購、結帳行為，即時更新推薦引擎。每秒可處理百萬筆事件，支撐大促（如雙11）期間的流量峰值。

🔬
微服務事件驅動架構
Netflix · Spotify · LINE

取代微服務間的同步 REST 呼叫，改用非同步事件溝通（Event-Driven Architecture）。解耦服務依賴，提高系統彈性。Saga Pattern 透過 Kafka 協調分散式交易。

🚗
IoT 與即時監控
Tesla · 台積電（TSMC）· GE

工廠感測器、車輛 GPS、設備監控資料統一串入 Kafka，後接 Flink / Spark Streaming 進行即時異常偵測與告警。

📝
活動日誌聚合（Log Aggregation）
所有大型系統

從多台伺服器收集 Application Log、Audit Log，集中送入 Kafka，再由 Logstash/Fluentd 消費送至 Elasticsearch，支撐 ELK 監控架構（現代 SIEM 的基石）。

業界架構模式：CQRS + Event Sourcing 搭配 Kafka — 將所有狀態變更以事件形式儲存在 Kafka，透過重播（Replay）重建任意時間點的狀態，是現代 DDD（Domain-Driven Design）的核心實踐。
07

Kafka vs 其他訊息系統

特性 Apache Kafka RabbitMQ AWS SQS Redis Streams
定位 事件串流平台 傳統 Message Broker 雲端托管 MQ 輕量串流
訊息保留 持久化，可重播 消費後刪除 14 天上限 記憶體為主
吞吐量 每秒百萬級 每秒萬~十萬 高（雲端彈性） 中等
延遲 毫秒級 微秒~毫秒 毫秒~數十毫秒 微秒級
消費模式 Pull（Consumer 主動拉） Push（Broker 推送） Pull / Long Poll Pull / XREAD
適合場景 大規模串流、資料管道 複雜路由、RPC AWS 生態整合 簡單任務佇列
選型建議：需要高吞吐量、訊息重播、多消費者群組 → 選 Kafka。需要複雜的訊息路由（Exchange、Binding）→ 選 RabbitMQ。全在 AWS 且不想維護 → 選 SQS/Kinesis。已有 Redis 且資料量不大 → Redis Streams 足夠。
