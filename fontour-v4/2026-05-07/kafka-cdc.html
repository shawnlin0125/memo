Kafka CDC 讀寫分離

利用 Change Data Capture 模式，透過 Kafka 達成資料庫讀寫分離，提升系統可擴展性與效能

核心概念
🔄

CDC — Change Data Capture

捕捉資料庫的每一筆變更（INSERT / UPDATE / DELETE），以事件串流形式發佈出去

⚡

Apache Kafka

高吞吐量的分散式訊息佇列，作為 CDC 事件的傳輸通道與緩衝層

✍️

寫入端（Write DB）

主資料庫（Primary），接受所有寫入操作，透過 binlog 產生 CDC 事件

📖

讀取端（Read Store）

訂閱 Kafka Topic 後自動同步，可為 Replica DB、Elasticsearch、Redis 等

系統架構流程圖
CDC 事件格式（Debezium JSON）
debezium-cdc-event.json
// Kafka Topic: db.inventory.orders { "before": { "id": 1001, "status": "pending" }, "after": { "id": 1001, "status": "shipped", "updated_at": 1700000000000 }, "source": { "db": "inventory", "table": "orders", "ts_ms": 1700000000000, "lsn": 12345 // PostgreSQL WAL offset }, "op": "u", // c=insert, u=update, d=delete, r=read(snapshot) "ts_ms": 1700000001234 // Debezium 處理時間戳 }
實作步驟
1

啟用資料庫 CDC 功能

MySQL 開啟 binlog（binlog_format=ROW）；PostgreSQL 設定 wal_level=logical 並建立 Replication Slot

2

部署 Debezium Connector

透過 Kafka Connect 平台部署 Debezium Connector，設定連線資訊、監控的 database/table、以及目標 Kafka Topic 命名規則（{db}.{schema}.{table}）

3

設計 Kafka Topic 分區策略

根據資料量與消費者數量規劃 Partition 數量。以主鍵作為 Partition Key，確保同一筆資料的事件順序一致，避免消費端亂序

4

實作 Consumer（讀取端同步）

針對不同讀取端（Replica DB、Elasticsearch、Redis）分別建立 Consumer Group，消費同一 Topic 但各自處理。實作冪等性（Idempotent）確保重複消費不會造成資料異常

5

監控與告警

監控 Consumer Lag（消費延遲），設定閾值告警。使用 Kafka Exporter + Prometheus + Grafana 建立可視化儀表板

Consumer 程式碼範例（Java / Spring Kafka）
OrderCdcConsumer.java
@Component public class OrderCdcConsumer { @KafkaListener( topics = "inventory.public.orders", groupId = "elasticsearch-sync-group" ) public void consume(ConsumerRecord<String, String> record) { DebeziumEvent event = parse(record.value()); switch (event.getOp()) { case "c": // INSERT esService.index(event.getAfter()); break; case "u": // UPDATE esService.update(event.getAfter()); break; case "d": // DELETE esService.delete(event.getBefore().getId()); break; } } }
傳統讀寫分離 vs Kafka CDC 讀寫分離
比較項目 傳統主從複寫 Kafka CDC
同步方式 資料庫原生複寫協定 事件串流（Event Streaming）
讀取端類型 僅能是相同類型資料庫 任意系統（ES、Redis、DW...）
延遲 毫秒～秒級 毫秒級（低延遲）
資料轉換 困難，需額外 ETL 可在 Consumer 端自由轉換
可擴展性 受資料庫 Replication 限制 水平擴展，支援多 Consumer Group
事件歷史 無法保存 Kafka 可保存完整 Changelog
複雜度 低 中～高（需維護 Kafka 叢集）
優缺點分析

✅ 優點

  • ＋讀寫完全解耦，主庫只承受寫入壓力
  • ＋支援多種讀取端（DB、ES、Redis、DW）
  • ＋Kafka 保存 Changelog，支援重播（Replay）
  • ＋Consumer 可依需求獨立擴縮容
  • ＋事件驅動架構，易於整合其他微服務
  • ＋低延遲同步，接近即時

❌ 缺點 / 注意事項

  • －最終一致性，讀取端可能有短暫延遲
  • －需維運 Kafka 叢集，增加運維複雜度
  • －Consumer 需處理冪等性（Idempotent）
  • －Schema 變更需同步更新所有 Consumer
  • －Debezium 與 DB 版本相依性管理
  • －初始快照（Snapshot）期間效能影響
適用場景
🔍

搜尋引擎同步

資料寫入 DB 後，自動同步至 Elasticsearch，實現近即時的全文搜尋

⚡

快取自動失效

DB 資料變更時，透過 CDC 事件自動清除或更新 Redis 快取，無需應用程式介入

📊

資料倉儲同步

將線上交易資料即時同步至分析型資料倉儲（Redshift、BigQuery），支援即時 BI 報表

🔗

微服務解耦

服務間透過 CDC 事件通訊，避免直接 API 耦合，提升系統彈性與容錯能力
