01 · Core Goal

核心目標

「新增一個票池 Node，不另開專案，開發完自動有 K8s 部署 config。」
Spring Boot 實現方式：Maven Multi-Module Monorepo + provider.yaml 身份宣告 + Helm Chart 模板 + CI/CD 自動渲染。
K8s / Helm 層與語言完全無關，只有建構工具和健康檢查端點不同。

整體架構由四個層次組成，各層職責清晰分離：

策略Monorepo
所有 Provider Node、Gateway、共用 Java 介面、Infra 設定放在同一個 Git Repo。透過 Maven Multi-Module 管理，根目錄一個 parent pom.xml 統一所有子模組。
策略Convention over Configuration
約定大於配置。只要 Node 的 Maven module 結構遵守規範、provider.yaml 格式正確，CI 就能自動處理 build、image 打包、Helm 渲染，不需每次手動設定。
策略Plugin 架構
每個 Provider Node 實作 IProviderNode Java interface 即可被 Gateway 識別。新增插件不需修改 Gateway 或主系統，做到真正的插拔式擴展。
Architecture Pattern
API Gateway + 插件式微服務 — 對外是一個子系統，對內是多個獨立部署的 Spring Boot 服務。
新增供應商 = 複製 template → 填 provider.yaml → 實作四個方法 → push → 自動部署。
02 · Glossary

名詞解釋

建構工具（Spring Boot 生態）

工具Maven Multi-Module
Spring Boot 原生的 Monorepo 方案。根目錄的 parent pom.xml 宣告所有子模組並統一管理依賴版本（BOM），各 Node 是獨立的 <module>，可單獨 build 或部署。
工具Gradle Multi-Project
Maven 的替代方案，build 速度更快。根目錄 settings.gradle 宣告所有子專案，各 Node 有自己的 build.gradle。適合對建構效能要求較高的大型 Monorepo。
工具Jib Plugin
Google 出品的 Maven/Gradle plugin。不需要寫 Dockerfile，直接 ./mvnw jib:build 就把 Spring Boot JAR 打包成 Docker image 推送到 Registry，並自動處理分層快取以加速 CI。
工具Maven BOM
Bill of Materials。在 parent pom.xml 的 <dependencyManagement> 統一鎖定所有子模組的依賴版本，避免各 Node 用到不同版本的同一個 library 造成衝突。
工具-pl / -am 旗標
-pl packages/viator-node 指定只建構特定 module，-am（also-make）自動包含它依賴的 shared module。CI 用 git diff 偵測有變動的 module 後，只 build 那個 Node，其他 Node 完全不動。

K8s / 部署

K8sHelm Chart
K8s 的部署模板引擎。把 Deployment、Service、HPA 等 YAML 參數化，透過 provider.yaml 填值後渲染成完整 manifest。Spring Boot 和其他語言的 Node 共用同一套 Helm Chart，語言無關。
K8sprovider.yaml
本架構設計的「Node 身份宣告檔」。每個 Node 只需填這一個檔案（name、image、resources、HPA 設定），CI 讀取後自動渲染 Helm manifest，不需手動寫任何 K8s YAML。
K8sHPA
Horizontal Pod Autoscaler。根據 CPU 使用率或自訂 metric 自動增減 Pod 數量。各 Node 在 provider.yaml 裡獨立設定 minReplicas / maxReplicas，互不影響。
K8sReadiness Probe
K8s 判斷 Pod 是否「準備好接收流量」的健康檢查。Spring Boot Actuator 內建 /actuator/health/readiness，加入 spring-boot-starter-actuator 依賴即自動啟用，無需自行實作。
K8sLiveness Probe
K8s 判斷 Pod 是否「還活著」的健康檢查。Spring Boot Actuator 內建 /actuator/health/liveness，失敗時 K8s 自動重啟 Pod。兩個 Probe 路徑需在 Helm Chart 的 provider.yaml 中宣告。
K8sinitialDelaySeconds
Probe 開始檢查前的等待秒數。Spring Boot 因 JVM 啟動需要時間，建議設為 30。設太低會導致 Pod 尚未啟動就被判定失敗而反覆重啟，造成 CrashLoopBackOff。
K8sRolling Update
K8s 預設更新策略，逐一替換舊 Pod，確保服務不中斷。搭配 Readiness Probe 可做到真正的零停機部署。Spring Boot 需額外在 application.yml 設定 server.shutdown: graceful。

可靠性（SRE）

SRECircuit Breaker
熔斷器。Spring Boot 生態使用 Resilience4j（取代已停止維護的 Hystrix）。外部 API 連續失敗超過閾值時，自動「斷路」並切換至 fallback 方法，防止故障蔓延。
SREResilience4j
Spring Boot 官方推薦的容錯函式庫。提供 Circuit Breaker、Retry、Rate Limiter、Bulkhead 四種機制，透過 annotation 宣告式設定，並與 Spring Boot Actuator 整合自動暴露狀態 metric 至 Prometheus。
SREBulkhead 艙壁隔離
Resilience4j 的 @Bulkhead annotation。限制每個 Node 的並發呼叫數（執行緒池或 semaphore 模式），Klook Node 的執行緒耗盡不會影響 KKday Node，實現程序層的故障隔離。
SREGraceful Shutdown
Spring Boot 2.3+ 內建支援。在 application.yml 設定 server.shutdown: graceful 與 spring.lifecycle.timeout-per-shutdown-phase: 30s，搭配 K8s terminationGracePeriodSeconds: 30，確保現有請求處理完畢再退出。
SRESpring Boot Actuator
Spring Boot 的生產就緒監控模組。自動暴露 /actuator/health、/actuator/metrics、/actuator/prometheus 等端點，與 Prometheus + Grafana 整合完全無需額外程式碼，加入依賴即啟用。
SREExternal Secrets Operator
K8s 的 Secret 管理方案。從 Vault 或 AWS Secrets Manager 自動同步 API Key 到 K8s Secret，Spring Boot 透過環境變數注入（@Value("${provider.api-key}")），避免敏感資訊進入 Git 或 provider.yaml。

共用介面（Java）

JavaIProviderNode interface
定義在 shared/interfaces Maven module 中的 Java interface，宣告 search()、createOrder()、confirmPayment()、cancel() 四個標準方法。各 Node 的 Spring Service 實作此 interface。
Java@Service annotation
Spring 的 Bean 宣告 annotation。每個 Node 的 Provider 實作類加上 @Service，Spring 容器自動管理其生命週期，並可透過建構子注入在 Gateway 中使用，無需手動實例化。
03 · Standard Operating Procedure

新增票池 Node 的完整 SOP

從複製 template 到自動部署完成，開發者只需執行以下步驟，不需接觸任何 K8s YAML 或 Helm 設定。

1
複製 _template module
從 Monorepo 的 packages/_template 複製一份作為新 Node 的起始點，包含標準的 pom.xml、目錄結構與 provider.yaml 範本。
bash 
cp -r packages/_template packages/viator-node
2
在根 pom.xml 宣告新 module
把新 Node 加入根目錄的 parent POM，Maven 才能識別這個子模組並在 build 時包含它。
pom.xml（根目錄） 
<modules>
  <module>shared/interfaces</module>
  <module>packages/gateway</module>
  <module>packages/klook-node</module>
  <module>packages/kkday-node</module>
  <module>packages/viator-node</module>  <!-- 新增 -->
</modules>
3
填寫 provider.yaml
唯一需要手填的部署設定。填完後 CI 自動渲染所有 K8s manifest，不需手動寫任何 YAML。注意 Spring Boot 的 initialDelaySeconds 要設 30 秒。
packages/viator-node/provider.yaml 
provider:
  name: viator
  image: your-registry/viator-node
  port: 8080

resources:
  requests:
    cpu: "200m"
    memory: "512Mi"
  limits:
    cpu: "1000m"
    memory: "1Gi"

hpa:
  minReplicas: 2
  maxReplicas: 5
  cpuTargetUtilization: 70

health:
  livenessPath: /actuator/health/liveness
  readinessPath: /actuator/health/readiness
  initialDelaySeconds: 30  # JVM 啟動較慢，必填
  gracePeriod: 30
4
實作四個標準介面方法
繼承 shared/interfaces 的 IProviderNode，加上 @Service 和 Resilience4j annotation。Gateway 和主系統零改動。
ViatorProvider.java 
@Service
public class ViatorProvider implements IProviderNode {

  @CircuitBreaker(name = "viator", fallbackMethod = "searchFallback")
  @Bulkhead(name = "viator")
  public SearchResult search(SearchRequest req) {
    // 呼叫 Viator 外部 API
  }

  @CircuitBreaker(name = "viator")
  public OrderResult createOrder(OrderRequest req) { ... }

  public ConfirmResult confirmPayment(ConfirmRequest req) { ... }

  public CancelResult cancel(CancelRequest req) { ... }

  // Fallback：熔斷時回傳 Redis 快取結果
  private SearchResult searchFallback(SearchRequest req, Exception e) {
    return cacheService.getCachedResult(req);
  }
}
5
推 PR → CI 自動 build 並 dry-run
CI 偵測有變動的 module，只 build 該 Node 並用 Jib 打 image，接著 helm template --dry-run 把 K8s manifest diff 貼到 PR comment，reviewer 直接看到會產生哪些資源。
.github/workflows/ci.yaml（節錄） 
# 偵測有變動的 module
CHANGED=$(git diff --name-only origin/main \
  | grep '^packages/' | cut -d'/' -f2 | sort -u)

# 只 build 有變動的 Node
./mvnw package -pl packages/$CHANGED -am -DskipTests=false

# Jib 打包並推送 image（不需 Dockerfile）
./mvnw jib:build -pl packages/$CHANGED \
  -Djib.to.tags=$GIT_SHA

# Helm dry-run，結果貼至 PR comment
helm template $CHANGED ./infra/charts/provider-node \
  --values packages/$CHANGED/provider.yaml \
  --set image.tag=$GIT_SHA \
  --dry-run
6
Merge → 自動部署
CI 渲染 Helm manifest 並 apply 至 ticket-middleware namespace，等待 Readiness Probe 通過後標記 Grafana deployment annotation。其他 Node 完全不動。
.github/workflows/deploy.yaml（節錄）
helm upgrade --install $CHANGED-node \ ./infra/charts/provider-node \ --values packages/$CHANGED/provider.yaml \ --set image.tag=$GIT_SHA \ --namespace ticket-middleware \ --create-namespace # 等待 Readiness Probe 通過 kubectl rollout status deployment/$CHANGED-node \ -n ticket-middleware # 標記 Grafana deployment annotation curl -X POST $GRAFANA_API/annotations \ -d '{"text":"Deploy: '"$CHANGED"'-node @ '"$GIT_SHA"'"}'
SOP 總結
開發者只需做三件事：複製 template、填 provider.yaml、實作四個方法。
CI/CD 自動處理：build → Jib 打包 → Helm 渲染 → K8s 部署 → Grafana 標記。
全程不需要接觸任何 K8s YAML，其他 Node 的服務不受影響。
